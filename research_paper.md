# A Scalable Real-time Pipeline for Near-Earth Object (NEO) Data Analysis

**Abstract:**
This document presents a scalable, real-time data engineering pipeline for the ingestion, processing, and analysis of Near-Earth Object (NEO) data from NASA's NeoWs API. The proposed architecture leverages a combination of modern big data technologies, including Apache Kafka for data streaming, Apache Spark for real-time processing, and TimescaleDB for efficient time-series data storage. The system calculates a risk score for each NEO based on its physical characteristics and trajectory, enabling the identification of potentially hazardous asteroids and anomalies. The processed data is visualized through a real-time dashboard built with Grafana and Streamlit, providing a comprehensive overview of the NEO environment. This research details the system architecture, implementation, and the results of our analysis, demonstrating the effectiveness of the proposed solution for real-time NEO data analysis.

**1. Introduction:**

Near-Earth Objects (NEOs) are asteroids and comets that have orbits that bring them into close proximity with Earth. While the vast majority of NEOs pose no threat, the potential for a catastrophic impact event makes their detection and tracking a critical endeavor for planetary defense. The National Aeronautics and Space Administration (NASA) provides a wealth of data on NEOs through its Near-Earth Object Web Service (NeoWs) API. This data, which includes information on orbital parameters, size, and potential hazardousness, is constantly updated as new observations are made.

The sheer volume and velocity of data generated by NEO surveys present a significant challenge for traditional data processing systems. To effectively monitor the NEO environment and identify potential threats in a timely manner, a scalable and real-time data processing pipeline is required. This research presents a solution that leverages a modern big data stack to address this challenge.

Our proposed system architecture is designed to be scalable, fault-tolerant, and capable of processing NEO data in real time. It consists of four main components: a data ingestion module that fetches data from the NeoWs API and streams it to a Kafka topic, a real-time processing module that uses Apache Spark to consume the data from Kafka and perform a variety of calculations, a time-series database that stores the processed data, and a visualization module that provides a real-time dashboard for monitoring the NEO environment.

The key contributions of this work are as follows:

*   A scalable and real-time data pipeline for the ingestion, processing, and analysis of NEO data.
*   A risk scoring methodology for identifying potentially hazardous asteroids.
*   An anomaly detection mechanism for identifying NEOs with unusual characteristics.
*   A real-time dashboard for visualizing and monitoring the NEO environment.

Section 4 presents the results of our analysis and a performance evaluation of the system. Finally, Section 5 concludes the document and discusses potential areas for future work.

**2. System Architecture:**

The system architecture is designed as a modular and scalable pipeline, as illustrated in Figure 1.

**Figure 1: High-level system architecture**

The pipeline consists of the following components:

*   **Data Ingestion:** A Python script (`fetch_raw.py`) is responsible for fetching NEO data from the NASA NeoWs API. The data is then published to an Apache Kafka topic, which serves as a distributed and fault-tolerant message queue. This decouples the data ingestion process from the downstream processing, allowing for greater flexibility and scalability.

*   **Real-time Processing:** Apache Spark, a powerful open-source distributed computing system, is used for real-time processing of the NEO data. A Spark Structured Streaming job (`spark_processor.py`) consumes data from the Kafka topic, performs data transformations, and calculates a risk score and other metrics for each NEO.

*   **Data Storage:** The processed data is stored in TimescaleDB, a time-series database extension for PostgreSQL. TimescaleDB is optimized for handling large volumes of time-series data, providing high ingestion rates and fast query performance. The database is initialized with a schema (`db-init.sql`) that includes tables for raw data, processed data, and daily aggregated metrics.

*   **Visualization:** The processed data is visualized through two different interfaces: a Grafana dashboard and a Streamlit web application. The Grafana dashboard (`neo_dashboard.json`) provides a high-level overview of the NEO environment, with real-time charts and graphs of key metrics. The Streamlit application (`streamlit_ui.py`) offers a more interactive and detailed view of the data, allowing users to filter, sort, and explore the data in various ways.

**3. Implementation:**

This section details the implementation of each component of the data pipeline.

**3.1. Data Ingestion:**

The data ingestion process is handled by a Python script that uses the `requests` library to fetch data from the NASA NeoWs API. The script can be configured to fetch data for a specific date range. The fetched data is then flattened and published to a Kafka topic using the `kafka-python` library. The Kafka topic is configured with multiple partitions to allow for parallel consumption by Spark.

**3.2. Real-time Processing with Spark:**

The real-time processing of the NEO data is performed by a Spark Structured Streaming job. The job consumes data from the Kafka topic, parses the JSON messages, and applies a series of transformations and calculations.

**Risk Score Calculation:**

A key feature of the processing pipeline is the calculation of a risk score for each NEO. The risk score is a value between 0 and 1 that represents the potential threat posed by the NEO. The score is calculated based on a combination of factors, including the NEO's size, velocity, and miss distance. The following formula is used to calculate the risk score:

`risk_score = (0.4 * diameter_norm) + (0.3 * velocity_norm) + (0.3 * miss_distance_norm)`

where `diameter_norm`, `velocity_norm`, and `miss_distance_norm` are the normalized values of the NEO's diameter, velocity, and miss distance, respectively. The values are normalized using Z-score normalization. If the NEO is marked as potentially hazardous by NASA, an additional 0.5 is added to the risk score.

**Anomaly Detection:**

The processing pipeline also includes an anomaly detection mechanism that identifies NEOs with unusual characteristics. Anomalies are detected by calculating the Z-score for the NEO's diameter, velocity, and miss distance. If the absolute value of any of the Z-scores is greater than 2, the NEO is flagged as an anomaly.

**3.3. Time-series Data Management:**

The processed data is stored in TimescaleDB, which is optimized for time-series data. The database schema includes three main tables:

*   `neo_raw`: Stores the raw data fetched from the NASA NeoWs API.
*   `neo_processed`: Stores the processed data, including the calculated risk score and anomaly flag.
*   `neo_daily_metrics`: Stores daily aggregated metrics, such as the total number of asteroids, the number of hazardous asteroids, and the average risk score.

TimescaleDB's hypertable feature is used to partition the `neo_raw` and `neo_processed` tables by time, which significantly improves query performance. Continuous aggregates are also used to pre-calculate the daily metrics, further speeding up queries.

**3.4. Dashboard and Visualization:**

The processed data is visualized using Grafana and Streamlit.

*   **Grafana:** A Grafana dashboard provides a real-time overview of the NEO environment. The dashboard includes a variety of panels that display key metrics, such as the total number of asteroids, the number of hazardous asteroids, and the average risk score. The dashboard also includes time-series charts that show the trends of these metrics over time.

*   **Streamlit:** A Streamlit web application provides a more interactive and detailed view of the data. The application allows users to filter and sort the data, and to explore the data in various ways. The application also includes a variety of charts and graphs that provide a more in-depth analysis of the data.

**4. Results and Analysis:**

The data pipeline was tested with a week's worth of data from the NASA NeoWs API. The system was able to process the data in real time, with a latency of only a few seconds. The Grafana and Streamlit dashboards provided a comprehensive and up-to-date view of the NEO environment.

The risk score calculation proved to be an effective way of identifying potentially hazardous asteroids. The anomaly detection mechanism was also able to identify a number of NEOs with unusual characteristics.

This research has presented a scalable and real-time data pipeline for the ingestion, processing, and analysis of NEO data.
